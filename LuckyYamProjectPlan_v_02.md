# LuckyYum — 전체 설계 & 실행 계획 (v2)

화면에 항상 떠 있는(오버레이) 내 스마트폰 속 사주 펫. 핵심 룰: **밥을 잘 주면 좋은 운세, 방치하면 나쁜 운세.**
실행 환경: Google Antigravity IDE (agent-first 워크플로우 기준)

> v1 → v2 변경 요약: ① 서버 제거(완전 온디바이스 + 정적 CDN), ② 등급 시프트 수식에 "돌봄 바닥 보장" 추가로 돌봄 우선 원칙을 수식으로 관철, ③ 첫 탭 락에 오탭 구제 규칙, ④ 알림을 AlarmManager 기준으로 수정, ⑤ M0-1에 삼성 실기 서비스 생존 검증 추가, ⑥ MBTI 4축을 참여도와 독립되게 정규화, ⑦ 죽음 페널티를 대흉에서 상실+재뽑기로 교체.

## Part 0. 확정된 설계 원칙 6가지

논쟁 끝난 전제. 이걸 흔들면 앱을 갈아엎어야 한다.

1. **2축 분리.** 운세의 세기(대길~대흉)와 결(영역·말투)은 서로 다른 축이다.
   - 세기 ← 돌봄 상태 (게임의 핵심)
   - 결 ← MBTI 성격 (차별화 한 스푼)
   - MBTI는 세기를 절대 안 건드린다. 안 그러면 돌봄 인과가 흐려진다.
2. **사주 = 출발점, 돌봄 = 결과를 뒤집는 힘.** 사주는 오늘의 베이스라인 등급만 정하고, 최종 결과는 돌봄이 더 세게 민다. **돌봄 우수 시 최소 등급 3(평)을 바닥으로 보장**해 "잘 키우면 절대 불운하지 않다"를 숫자로 관철한다(1-C).
3. **실시간 LLM 없음.** Gemini는 출시 전 콘텐츠 공장으로만. 운세는 전부 사전 생성해 조회만.
4. **서버 없음(데모 기준).** 콘텐츠 DB는 SQLite로 앱에 번들. 갱신은 CDN 정적 파일 버전 체크 다운로드. 펫 상태·care_log·MBTI 집계 전부 온디바이스. 서버는 궁합 공유·과금·랭킹이 필요해지는 시점에 그때 세운다.
5. **생시는 기기 밖으로 절대 안 나간다.** 사주코드조차 전송하지 않는다(사주 8자는 생년월일시를 사실상 역산 가능 → 익명정보가 아니다). 서버가 없으므로 이 원칙은 아키텍처 수준에서 자동 보장된다.
6. **Android 전용은 선택이 아니라 제약이다.** 오버레이(SYSTEM_ALERT_WINDOW)는 iOS에 대응 개념이 없다. iOS는 "나중에"가 아니라 "오버레이 컨셉으로는 불가능"이며, 확장 시 위젯/라이브 액티비티 기반 별도 컨셉이 필요하다는 것을 전제로 깔고 간다.

## Part 1. 운세 생성 파이프라인 (최종본)

### 1-A. 오프라인 (콘텐츠 공장 — 출시 전 / 주기 배치)

```
사주 조합 공간 정의
   └─ 대표 조합 N개(수백~600+)  ─▶  fortune_base (SQLite 번들)
                                     (saju_code → 베이스라인 등급 + 강조 카테고리)

조립식 문구 생성 (Gemini 배치)
   [등급 문장 5종] × [MBTI 말투 16] × [카테고리 태그]
        └─ 전량 곱하지 않고 "조립"으로 커버  ─▶  fortune_text (SQLite 번들)
        └─ 사람 검수 후 적재 (헛소리·중복 필터)

번들 빌드 스크립트
   └─ 검수 통과분 → content_vX.db 생성 → 앱 에셋 포함 / CDN 업로드
```

핵심: 조립식이라 16성격을 손으로 다 안 쓴다. 최종 문구 = [세기 등급 문장] + [MBTI 말투 변형] + [영역 카테고리]. 사주 조합은 "베이스라인 등급 + 강조 카테고리"만 매핑하면 된다. 전체 콘텐츠 DB는 수천 행, 수 MB — 번들에 넉넉히 들어간다.

### 1-B. 런타임 (100% 온디바이스, 네트워크 불필요)

```
1. 생년월일시 → 만세력 → 오늘의 사주코드          ※ 기기 밖 전송 없음
2. 사주코드 → fortune_base(로컬 SQLite) → 베이스라인 등급(1~5)
3. 포만감·친밀도 스냅샷 → 등급 시프트              ← 게임의 심장
4. 그날 첫 탭에 최종 등급 확정 → 하루 고정 (락)    ← 오탭 구제 1회 (1-C)
5. 최종등급 × MBTI × 카테고리 → fortune_text 조회
6. 펫이 자기 MBTI 말투로 읽어줌
```

### 1-C. 등급 시프트 수식 (돌봄이 사주를 이기게 — 수식으로 보장)

```
baseline (1~5)  ← 사주

care_shift:
    포만감>70 & 친밀도>60  → +2      # 우수 돌봄
    포만감 30~70           →  0
    포만감<30              → −2
    포만감=0 또는 방치      → −3

final = clamp(baseline + care_shift, 1, 5)

바닥 보장 (핵심):
    우수 돌봄 조건 충족 시 final = max(final, 3)
    → 문구: "네 정성이 오늘의 액운을 막았어."   # 밥 줄 이유 그 자체

락 후 구제 (오탭 방어):
    락 이후 그날 첫 밥주기 1회에 한해 final +1 (상한 5)
    → 문구: "늦었지만… 고마워. 조금 나아질 것 같아."
```

- 보상 진폭(+2, 바닥 3)과 처벌 진폭(−3)이 baseline 격차(±2)를 이긴다 → "잘 키우면 행운, 방치하면 불행"이 모든 baseline에서 성립. baseline 1(대흉)인 날도 우수 돌봄이면 최소 평(3), 방치면 어떤 사주도 대흉(1).
- 랜덤 항은 뺀다. 인과가 게임의 전부인데 노이즈를 넣을 이유가 없다. 변동성은 사주 baseline이 날마다 바뀌는 것으로 충분하다.
- 알·유년기엔 final을 흐리게(신뢰도 낮게) 표시해 성장 동기를 만든다.

### 1-D. 성장 단계별 운세 응답

| 단계 | 운세 응답 |
|---|---|
| 알 | 응답 없음 ("아직 잠들어 있어요") |
| 아기 | 어설픈 대답 (등급 흐릿, 한 줄) |
| 유년기/청소년 | 부분 응답 (카테고리 1개) |
| 성체 | 완전 응답 (MBTI 말투 + 전체 카테고리) |

알람은 하루 1회만 설정 가능. 정해진 시각에 그날 운세 1회 전달 (구현은 Part 3-알림 참조 — WorkManager 아님).

## Part 2. 펫 육성 & MBTI 시스템

### 2-A. 성장 로직

```
알 뽑기(가챠, 데모 10종) → 알 → 아기 → 청소년 → 성체
                                          └─ 성체 전환 시 MBTI 확정 & 고정
```

- 상태값: 포만감, 친밀도 (시간 경과 하락 — 마지막 저장 시각 역산, 백그라운드 타이머 없음)
- 각 단계 전환은 누적 돌봄량 + 경과 시간 조건
- **죽음 규칙(v2 변경):** 아기 때 죽으면 별이 되어 도감에 영구 기록되고, **즉시 새 알 뽑기가 열린다.** 그날 운세 대흉 페널티는 폐지. 이유: 펫을 죽게 둔 유저는 이미 이탈 경계에 있는 유저다. 상실(도감의 별)은 이미 충분한 페널티고, 여기에 대흉까지 때리면 복귀가 아니라 삭제를 부른다. 죽음은 처벌이 아니라 재시작 훅이다.

### 2-B. MBTI 4축 집계 (참여도와 독립된 정규화 — v2 재설계)

**v1의 결함:** E/I(빈도)·S/N(새것 시도)·J/P(규칙성)가 전부 헤비유저 쪽으로 쏠려, 헤비유저는 EN?J, 라이트유저는 IS?P로 수렴 → 사실상 16종이 아니라 2~3종만 나온다. v2는 모든 축을 **유저 자신의 행동 내 비율/패턴**으로 판정해 총 사용량과 분리한다.

| 축 | 신호 (참여도 독립) | 한쪽 | 반대쪽 |
|---|---|---|---|
| E / I | **세션 스타일**: 총 상호작용이 같을 때, 짧고 잦은 세션 vs 길고 드문 세션 (세션 수 / 세션당 평균 행동 수의 비) | 잘게 자주 → E | 몰아서 깊게 → I |
| S / N | **새것 시도 비율**: 전체 행동 중 처음 하는 먹이·놀이·외출의 비율 | 반복 → S | 새것 → N |
| T / F | **행동 성질 비율**: 니즈해결(밥·청소·치료) vs 애정(쓰다듬·칭찬)의 비율 | 니즈 → T | 애정 → F |
| J / P | **돌봄 시각 규칙성**: 하루 중 돌봄 시각의 원형분산(circular variance) — 표본 수와 무관한 통계량 | 분산 낮음(규칙) → J | 분산 높음(즉흥) → P |

- 4축 모두 "얼마나 많이"가 아니라 "어떤 비율·리듬"만 본다. 헤비/라이트 유저가 같은 성격 분포를 가질 수 있다.
- **표본 미달 규칙(랜덤 금지):** 축별 최소 로그 수(예: 20) 미달 시 해당 축은 **잠정 판정**으로 성체 전환은 그대로 진행하되, 성체 후 14일간의 로그로 1회 자동 보정을 허용한다. "펫의 성격이 자리를 잡았어요" 이벤트로 서사화 — 데이터 부족을 기능처럼 보이게 만든다. 보정 후 영구 고정.

### 2-C. MBTI "한 스푼" (결·영역·말투만 변형 — v1 유지)

| 축 | 운세에 넣는 변형 |
|---|---|
| E / I | E: 대인·기회운 강조 / I: 내면·건강·집중운 |
| S / N | S: 당장의 구체적 실용 조언 / N: 큰 흐름·징조 |
| T / F | T: 냉정·전략 문체 / F: 공감·관계 문체 |
| J / P | J: 계획·마감 지침 / P: 즉흥·뜻밖의 기회 |

같은 "길(吉)"이라도 ESTJ 펫은 군기 잡듯, INFP 펫은 몽환적으로 읽는다. 이게 포스텔러·점신과 안 겹치는 유일한 차별점.

## Part 3. 오버레이 & 알림 (안드로이드)

### 오버레이

- **권한:** SYSTEM_ALERT_WINDOW. 자동 부여 불가 → `Settings.canDrawOverlays()` 확인 후 설정 화면으로 안내.
- **뷰:** 포그라운드 Service + `WindowManager.addView()`. `TYPE_APPLICATION_OVERLAY`, `FLAG_NOT_FOCUSABLE`, 투명 배경. 드래그 이동, 탭 → 확장.
- **포그라운드 서비스 타입(Android 14+):** `specialUse` 선언 + Play 심사용 사유 소명 문구 준비. 이건 코드가 아니라 심사 리스크 — 데모 단계에서 문구 초안까지 만들어 둔다.
- **생존성(이 앱의 사망 시나리오):** 삼성 One UI의 배터리 절전은 포그라운드 서비스를 죽인다. 한국 시장 = 삼성 과반. "며칠 뒤 펫이 조용히 사라져 있음"이 최악의 실패다. 대응:
  1. 배터리 최적화 제외 요청 플로우(`REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`) — 온보딩에서 펫의 대사로 요청("나 여기 계속 있고 싶어…").
  2. 서비스 사망 감지 → 재기동 로직 + 다음 앱 진입 시 "펫이 잠들어 있었어요" 복구 UX.
  3. **M0-1 검증 항목에 삼성 실기 24시간 생존 + Doze 진입/복귀 테스트 포함(필수).**
- **떠 있을 이유(리텐션 훅):** 오버레이 펫에 포만감 배지 + "밥 줄 시간" 표시 + 탭하면 그 자리에서 밥. 위젯형 미니 돌봄 UI여야 유저가 안 끈다.
- **탭 게이팅:** 굶었거나 방치 상태면 "…배고파서 오늘은 안 보여요" → 밥 줘야 운세 해금. 운세 자체를 보상으로.
- **게이팅과 락의 경계(명세):** 게이팅 상태(포만감<30)에서의 탭은 락을 트리거하지 않는다. 락은 "운세가 실제로 표시된 첫 순간"에만 걸린다. 게이팅 해제(밥) 직후의 탭이 락 탭이다. + 락 후 구제 1회(1-C).
- **시간 경과:** 백그라운드 타이머 ❌. 마지막 저장 시각 역산으로 스탯 감소 계산.

### 알림 (v2 수정 — WorkManager 폐기)

- WorkManager는 지연 허용 배치용이라 **정시 보장이 안 된다.** "정해진 시각의 운세 알림"은:
  1. 기본: `AlarmManager.setExactAndAllowWhileIdle()` + `SCHEDULE_EXACT_ALARM`/`USE_EXACT_ALARM` 권한 플로우 (Android 14+ 신규 설치는 기본 거부 → 온보딩에서 펫 대사로 요청).
  2. 폴백: 권한 거부 시 WorkManager 비정시 알림으로 다운그레이드하고 "±15분쯤 늦을 수 있어요"를 명시.
- 하루 1회 설정 제약은 유지.

## Part 4. 데이터 모델 (전부 온디바이스)

| 테이블/저장소 | 위치 | 주요 필드 |
|---|---|---|
| fortune_base | 번들 SQLite (읽기 전용) | saju_code, baseline_grade(1~5), emphasis_category |
| fortune_text | 번들 SQLite (읽기 전용) | grade, mbti, category, tone_variant, text |
| content_meta | 번들 SQLite | content_version (CDN 버전 체크용) |
| user_profile | 로컬 (암호화 저장) | birth_datetime(로컬만), alarm_time, exact_alarm_granted |
| pet_state | 로컬 Room | pet_id, stage, hunger, intimacy, mbti(확정 후), mbti_provisional_axes, last_saved_at |
| daily_lock | 로컬 Room | date, final_grade, redeemed(구제 사용 여부) |
| care_log | 로컬 Room | pet_id, action_type, is_novel(새것 여부), session_id, ts |
| memorial(도감) | 로컬 Room | pet_id, species, died_at_stage, ts |

- 콘텐츠 갱신: 앱 시작 시 CDN(Cloudflare 정적 호스팅)의 `content_meta.json` 버전 비교 → 신규면 `content_vX.db` 다운로드 교체. 서버 로직 0.
- 백업/기기이전은 데모 범위 밖(로컬 유실 감수). 정식판에서 Android Auto Backup 또는 계정 동기화로.

## Part 5. 기술 아키텍처 (v2 — 서버리스)

```
[React Native 앱]  ← 여기가 전부다
   ├─ 온디바이스 만세력 라이브러리 (사주코드 계산, 생시 기기 밖 전송 없음)
   ├─ 번들 SQLite: fortune_base / fortune_text (조회 전용)
   ├─ 펫 육성 루프 / 스탯 역산 / care_log / MBTI 4축 집계 (전부 로컬)
   ├─ 운세 락 & 구제 로직 (daily_lock)
   └─ Kotlin 네이티브 모듈
        ├─ 오버레이 (SYSTEM_ALERT_WINDOW / ForegroundService(specialUse) / WindowManager)
        └─ 정시 알림 (AlarmManager exact + 권한 플로우, WorkManager 폴백)

[Cloudflare 정적 호스팅 (Pages/R2)]
   └─ content_meta.json + content_vX.db   ※ 서버 코드 없음, 파일 서빙만

[Gemini 3 Flash/Pro]  ※ 오프라인 콘텐츠 공장 전용 (런타임 호출 없음)
   └─ fortune_text 대량 조립 생성 → 사람 검수 → 번들 빌드 스크립트 → SQLite
```

- FastAPI/PostgreSQL/Docker 전면 삭제(데모 기준). 얻는 것: 서버 비용 0, 완전 오프라인 동작, 지연 0, 생시 프라이버시의 구조적 보장, 운영 부담 0.
- 서버가 다시 필요해지는 트리거(그때 세운다): MBTI 궁합 공유 매칭, 과금 검증, 랭킹/소셜, 콘텐츠 A/B.
- RN + Kotlin 네이티브 오버레이 방향 유지. RN 기성 오버레이 라이브러리는 관리 안 되는 게 많으므로 직접 작성.

## Part 6. 실행 계획 — Antigravity 미션 단위 (v2)

운용 원칙 (v1 유지 + 보강):
- **Planning mode:** 만세력, 운세 파이프라인(락·구제 포함), MBTI 집계, 오버레이·알림 네이티브.
- **Fast mode:** 스캐폴딩·CRUD·보일러플레이트·콘텐츠 배치 스크립트.
- **Knowledge Item 시딩:** 이 v2 문서를 프로젝트 지식으로. 특히 "2축 분리", "바닥 보장 수식", "서버 없음", "MBTI 비율 정규화"는 에이전트가 어기기 쉬운 전제이므로 문서 상단 원칙 6개를 강조.
- **병렬화:** "콘텐츠 공장 / 앱 코어"는 독립 → Manager 뷰 동시 진행. (백엔드 트랙은 소멸)
- **에이전트가 못 하는 부분(명시):** 오버레이·알림·만세력은 실기(특히 삼성) 검증 필요. 에이전트는 코드까지, 실기 검증은 사람. Antigravity 브라우저 검증은 콘텐츠 검수 화면·번들 빌드 결과 확인에만 유효.
