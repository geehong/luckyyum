# LuckyYum — 전체 설계 & 실행 계획 (v1)

> 화면에 항상 떠 있는(오버레이) 내 스마트폰 속 사주 펫.
> 핵심 룰: **밥을 잘 주면 좋은 운세, 방치하면 나쁜 운세.**
> 실행 환경: **Google Antigravity IDE** (agent-first 워크플로우 기준)

---

## Part 0. 확정된 설계 원칙 4가지

이전 논의에서 잡은 보충 사항을 **논쟁 끝난 전제**로 못 박고 시작한다. 이걸 흔들면 나중에 앱을 갈아엎어야 한다.

1. **2축 분리.** 운세의 *세기*(대길~대흉)와 *결*(영역·말투)은 서로 다른 축이다.
   - 세기 ← **돌봄 상태** (게임의 핵심)
   - 결 ← **MBTI 성격** (차별화 한 스푼)
   - MBTI는 세기를 절대 안 건드린다. 안 그러면 돌봄 인과가 흐려진다.
2. **사주 = 출발점, 돌봄 = 결과를 뒤집는 힘.** 사주는 오늘의 베이스라인 등급만 정하고, 최종 결과는 돌봄이 더 세게 민다. 사주가 이기면 밥 줄 이유가 사라진다.
3. **실시간 LLM 없음.** Gemini는 "실시간 서비스"가 아니라 **출시 전 콘텐츠 공장**으로만 쓴다. 운세는 전부 사전 생성해 DB에서 조회만. (비용·지연·오프라인·검열 문제 동시 해결)
4. **개인정보 온디바이스.** 생년월일시는 민감정보(PIPA 대상). 사주 코드 계산은 앱에서 하고, **생시 원본은 서버로 안 보낸다.** 서버엔 익명 사주코드만.

## Part 1. 운세 생성 파이프라인 (최종본)

### 1-A. 오프라인 (콘텐츠 공장 — 출시 전 / 주기 배치)

```
사주 조합 공간 정의
   └─ 대표 조합 N개(수백~600+)  ─▶  fortune_base 테이블
                                     (saju_code → 베이스라인 등급 + 강조 카테고리)

조립식 문구 생성 (Gemini 배치)
   [등급 문장 5종] × [MBTI 말투 16] × [카테고리 태그]
        └─ 전량 곱하지 않고 "조립"으로 커버  ─▶  fortune_text 테이블
        └─ 사람 검수 후 적재 (헛소리·중복 필터)
```

**핵심: 조립식이라 16성격을 손으로 다 안 쓴다.**
최종 문구 = `[세기 등급 문장]` + `[MBTI 말투 변형]` + `[영역 카테고리]`. 사주 조합은 "베이스라인 등급 + 강조 카테고리"만 매핑하면 된다.

### 1-B. 런타임 (온디바이스 + 서버 조회, 실시간 LLM ❌)

```
1. (온디바이스) 생년월일시 → 만세력 → 오늘의 사주코드   ※ 원본 생시 서버 전송 X
2. (조회)       사주코드 → fortune_base → 베이스라인 등급(1~5)
3. (온디바이스) 포만감·친밀도 스냅샷 → 등급 시프트        ← 게임의 심장
4. (락)         그날 첫 탭에 최종 등급 확정 → 하루 고정
5. (조회)       최종등급 × MBTI × 카테고리 → fortune_text에서 완성 문구 조회
6. (표시)       펫이 자기 MBTI 말투로 읽어줌
```

### 1-C. 등급 시프트 수식 (돌봄이 사주를 이기게)

```
baseline (1~5)  ← 사주
care_shift:
    포만감>70 & 친밀도>60 → +1
    포만감 30~70          →  0
    포만감<30             → −2
    포만감=0 또는 방치     → −3
final = clamp(baseline + care_shift + small_random(-0..+0), 1, 5)
```

돌봄의 진폭(−3~+1)이 사주 baseline보다 크다 → "잘 키우면 행운, 방치하면 불행"이 숫자로 보장된다. 알·유년기엔 final을 흐리게(신뢰도 낮게) 처리해 성장 동기를 만든다.

### 1-D. 성장 단계별 운세 응답

| 단계 | 운세 응답 |
|---|---|
| 알 | 응답 없음 ("아직 잠들어 있어요") |
| 아기 | 어설픈 대답 (등급 흐릿, 한 줄) |
| 유년기/청소년 | 부분 응답 (카테고리 1개) |
| 성체 | 완전 응답 (MBTI 말투 + 전체 카테고리) |

알람은 **하루 1회만 설정 가능**. 정해진 시각에 그날 운세 1회 전달(WorkManager).

## Part 2. 펫 육성 & MBTI 시스템

### 2-A. 성장 로직

```
알 뽑기(가챠, 데모 10종) → 알 → 아기 → 청소년 → 성체
                                              └─ 성체 전환 시 MBTI 16종 중 하나 확정 & 고정
```

- **상태값**: 포만감, 친밀도 (시간 경과 하락)
- 각 단계 전환은 누적 돌봄량 + 경과 시간 조건
- 아기 때 죽으면 → MBTI 미확정 상태로 별이 됨, 그날 운세 대흉권 (성장 실패 = 페널티)

### 2-B. MBTI 4축 집계 (돌봄 패턴 로깅)

성장 내내 `care_log`에 행동을 기록하고, 성체 전환 순간 4축을 각각 임계값으로 판정한다. **"무슨 행동"이 아니라 "어떤 비율/리듬"**이 핵심.

| 축 | 신호 | 한쪽 | 반대쪽 |
|---|---|---|---|
| **E / I** | 총 상호작용 **빈도** (일평균 터치·세션) | 자주 → E | 가끔 → I |
| **S / N** | **새것 시도** 비율 (새 먹이·놀이·외출) | 반복 → S | 새것 → N |
| **T / F** | 니즈해결(밥·청소·치료) vs 애정(쓰다듬·칭찬) | 니즈 → T | 애정 → F |
| **J / P** | 돌봄 시간대 **규칙성** (시각 분산) | 규칙적 → J | 불규칙 → P |

E/I(빈도)와 T/F(성질)를 다른 지표로 분리한 게 포인트 — 겹치면 MBTI가 무너진다.

### 2-C. MBTI "한 스푼" (결·영역·말투만 변형)

| 축 | 운세에 넣는 변형 |
|---|---|
| E / I | E: 대인·기회운 강조 / I: 내면·건강·집중운 |
| S / N | S: 당장의 구체적 실용 조언 / N: 큰 흐름·징조 |
| T / F | T: 냉정·전략 문체 / F: 공감·관계 문체 |
| J / P | J: 계획·마감 지침 / P: 즉흥·뜻밖의 기회 |

같은 "길(吉)"이라도 ESTJ 펫은 군기 잡듯, INFP 펫은 몽환적으로 읽는다. 이게 포스텔러·점신과 안 겹치는 유일한 차별점.

## Part 3. 오버레이 설계 (안드로이드)

- **권한**: `SYSTEM_ALERT_WINDOW`. 자동 부여 불가 → `Settings.canDrawOverlays()` 확인 후 사용자를 설정 화면으로 보내 직접 ON.
- **뷰**: 포그라운드 `Service` + `WindowManager.addView()`로 펫 뷰 표시. `TYPE_APPLICATION_OVERLAY`, `FLAG_NOT_FOCUSABLE`, 투명 배경. 드래그 이동, 탭 → 확장.
- **떠 있을 이유(리텐션 훅)**: 오버레이 펫에 포만감 배지 + "밥 줄 시간" 알림 + 탭하면 그 자리에서 밥. 위젯형 미니 돌봄 UI여야 유저가 안 끈다.
- **탭 게이팅**: 굶었거나 방치 상태면 "…배고파서 오늘은 안 보여요" → 밥 줘야 운세 해금. **운세 자체를 보상으로** 만든다.
- **시간 경과**: 백그라운드 타이머 ❌. **마지막 저장 시각 역산**으로 스탯 감소 계산.
- **알림**: 운세 알림만 `WorkManager` 하루 1회 정시.

## Part 5. 기술 아키텍처 (확정)

```
[React Native 앱]
   ├─ 온디바이스 만세력 라이브러리 (사주코드 계산, 생시 원본 로컬 보관)
   ├─ 펫 육성 루프 / 스탯 역산 / MBTI 집계
   └─ Kotlin 네이티브 오버레이 모듈 (SYSTEM_ALERT_WINDOW / ForegroundService / WindowManager)
            │  (RN 브리지)
            ▼
[FastAPI + PostgreSQL]  (Docker Compose)
   ├─ /fortune/base   : saju_code → baseline 조회
   ├─ /fortune/text   : (grade, mbti, category) → 완성 문구 조회
   ├─ 펫 상태: in-memory dict → 주기 DB 저장 (다중 인스턴스 가면 Redis)
   └─ care_log 수집 + MBTI 집계 job

[Gemini 3 Flash/Pro]  ※ 오프라인 콘텐츠 공장 전용 (런타임 호출 없음)
   └─ fortune_text 대량 조립 생성 → 검수 → 적재
```

- RN + `SYSTEM_ALERT_WINDOW` 방향은 유지. **단, 오버레이 네이티브 모듈은 직접 Kotlin으로.** (RN 기성 오버레이 라이브러리는 관리 안 되는 게 많음)
- 사전 생성(Part 0-3) 덕에 초기 서버는 사실상 조회 API만. 부하 낮음.

# Part 6. 실행 계획 — Antigravity 미션 단위

Antigravity는 **Manager 뷰에서 여러 에이전트를 병렬로** 굴리고, 각 작업을 **구현계획 → 태스크리스트 → (코딩) → 브라우저/스크린샷 검증** 아티팩트로 내놓는다. 이 특성에 맞춰 아래처럼 운용한다.

### 운용 원칙

- **Planning mode(계획 검토 필수)**: 로직·정확도 리스크가 큰 구간. → 만세력, 운세 파이프라인, MBTI 집계, 오버레이 네이티브.
- **Fast mode(빠른 실행)**: 스캐폴딩·CRUD·보일러플레이트.
- **Knowledge Item 시딩**: 이 설계 문서를 프로젝트 지식으로 넣어 에이전트가 전제(2축 분리·돌봄 우선순위)를 안 어기게 한다.
- **병렬화**: "콘텐츠 공장 / 백엔드 / 앱코어"는 서로 독립 → Manager 뷰에서 동시 진행.
- **에이전트가 못 하는 부분(명시)**: 오버레이·온디바이스 만세력은 **실제 안드로이드 에뮬레이터/기기 검증 필요**. 에이전트는 코드까지, 실기 검증은 사람이. Antigravity 브라우저 검증은 백엔드 API·관리자 화면에만 유효.
