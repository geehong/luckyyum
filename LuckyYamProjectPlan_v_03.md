# LuckyYum — 전체 설계 & 실행 계획 (v3)

화면에 항상 떠 있는(오버레이) 내 스마트폰 속 사주 펫. 핵심 룰: **밥을 잘 주면 좋은 운세, 방치하면 나쁜 운세.**
실행 환경: Google Antigravity IDE (agent-first 워크플로우 기준)

> **v2 → v3 변경 요약 (하이브리드 & 크로스 플랫폼):** 
> ① `shorts100.com`의 검증된 백엔드/유저 템플릿을 재도입하여 **소셜 가입 및 랭킹 시스템** 추가 (리텐션 강화).
> ② 생년월일시(사주)는 기기에 남기고, 통계용 정보(나라/성별/나이대)만 서버로 보내 프라이버시 방어.
> ③ 향후 iOS 진출을 위해 코어 플레이를 **인앱(In-App)**으로 빼고, 안드로이드 오버레이는 선택적 토글(Toggle) 기능으로 분리.
> ④ 리뷰 반영 완료: Zustand 상태 관리, 게스트 모드, CDN 오프라인 패치, 도감 연동 및 4주 개발 일정 구체화.

## Part 0. 확정된 설계 원칙 6가지

1. **2축 분리:** 운세의 세기(대길~대흉)와 결(영역·말투)은 서로 다른 축이다. MBTI는 세기를 건드리지 않는다.
2. **바닥 보장 수식:** 사주는 베이스라인일 뿐, 돌봄 우수 시 최소 등급 3(평)을 바닥으로 보장한다.
3. **콘텐츠 오프라인 번들링:** 운세 텍스트는 서버 API를 매번 호출하지 않고, SQLite로 앱에 번들링하여 비용과 지연을 0으로 만든다. (앱 시작 시 CDN 버전 체크 후 OTA 업데이트)
4. **프라이버시 철벽 방어:** 생년월일시(사주코드)는 절대 서버로 보내지 않고 기기 내부에서만 연산한다. 서버에는 통계 데이터와 펫의 상태만 전송한다.
5. **검증된 백엔드 재활용:** 유저 인증 및 랭킹 시스템은 이미 마켓 심사를 통과한 `shorts100.com`의 템플릿(FastAPI+DB)을 100% 재활용하여 개발 공수와 리스크를 제거한다.
6. **오버레이와 인앱의 분리 (iOS 대비):** 모든 핵심 육성/운세 로직은 앱의 메인 화면(인앱)에서 완벽히 구동된다. 안드로이드 오버레이는 뷰(View) 역할만 하는 선택적 위젯으로 취급하여, 추후 iOS 포팅 시 오버레이만 끄면 되도록 아키텍처를 분리한다.

## Part 1. 아키텍처 및 폴더 구조 (하이브리드)

### 1-A. 전체 데이터 흐름

```text
[React Native 앱 (메인 코어)]
   ├─ 온디바이스 만세력 (사주 계산, 기기 내 보관)
   ├─ 번들 SQLite: fortune_base / fortune_text (운세 조회 전용)
   │    └─ 앱 구동 시 Cloudflare CDN에서 content_meta.json 비교 후 OTA 업데이트
   ├─ 펫 육성 루프 / 돌봄 수치 연산 / 인앱(In-App) 메인 화면 UI
   │    └─ 상태 관리: Zustand (인앱 로직) + MMKV/AsyncStorage (로컬 영속 저장)
   ├─ 안드로이드 네이티브 모듈
   │    ├─ 선택적 오버레이 (SYSTEM_ALERT_WINDOW / ON-OFF 토글 가능 / Zustand 단방향 구독)
   │    └─ 정시 알림 (AlarmManager)
   └─ 백엔드 통신 모듈 (유저 가입, 통계 전송, 랭킹 동기화)

[백엔드 서버 (FastAPI + PostgreSQL - shorts100 템플릿)]
   ├─ /auth    : 소셜 로그인(구글/애플), 세션 관리
   ├─ /users   : 국가, 성별, 나이대(Demographics) 수집 (통계/마케팅용)
   └─ /ranking : 펫 닉네임, 단계, MBTI, 돌봄 점수 동기화 및 리더보드 서빙 (시간 조작 어뷰징 방어 필터 포함)

[오프라인 콘텐츠 공장 (Gemini)]
   └─ 조립식 운세 텍스트 배치 생성 → 앱 빌드 시 SQLite로 말아넣음 (총 조합 200~600개)
```

## Part 2. 유저 가입 및 랭킹 시스템 (게스트 모드 필수화)

앱 설치 직후의 가입 마찰률을 0%로 만들기 위해 오프라인 ↔ 온라인 전환 구조를 가집니다.

### 2-A. 게스트 모드 (로컬 저장, 서버 전송 ❌)
*   **시작:** 최초 앱 실행 시 회원가입 없이 '생년월일/시간/성별'만 입력받아 **무조건 오프라인(게스트) 모드**로 즉시 시작.
*   **목적:** 온디바이스 만세력을 돌려 오늘의 베이스라인 등급을 찾고 코어 플레이를 바로 체험.

### 2-B. 랭킹 연동 (서버 전송 ⭕️)
*   **연동 트리거:** 유저가 '랭킹 보드' 탭에 진입 시 **"내 펫 랭킹 올리기 (소셜 로그인)"** 버튼을 통해 연동 유도.
*   **수집 정보:** 소셜 로그인 토큰(이메일), 국가, 나이대, 펫 닉네임, 펫 스탯. (생시는 절대 전송 안 함)
*   **랭킹 어뷰징 방어:** 랭킹은 1차적으로 재미 요소이나, 시간 조작(Time Travel) 방지를 위해 서버 수신 시 `last_care_time` 간격이 비정상적으로 짧은 경우 필터링(Ignore).

## Part 3. iOS 확장을 위한 오버레이 분리 로직

*   **상태의 단일 진실 공급원(SSOT):** 펫의 포만감, 친밀도 등의 상태는 RN 앱 내부의 `Zustand`가 메인이며, `MMKV/AsyncStorage`에 영구 저장됩니다.
*   **오버레이의 역할:** 안드로이드 오버레이 서비스는 RN 브릿지를 통해 Zustand 상태를 단방향 '구독'하여 그려주고, 밥 주기 등의 Action만 RN으로 쏴주는 껍데기(View) 역할만 합니다.
*   **크로스 플랫폼 대응:** 앱 내 설정에서 "항상 떠있기(오버레이 모드)" 토글을 끄면 평범한 인앱 다마고치가 됩니다. iOS 빌드 시에는 이 토글 자체를 비활성화하면 코어 로직 수정 없이 즉시 포팅이 가능합니다.

## Part 4. 펫 육성, 죽음(도감) & 운세 메커니즘

### 4-A. 육성 및 죽음(Memorial)
*   **돌봄 시프트:** 포만감>70 & 친밀도>60 이면 운세 등급 상승 (바닥 3등급 보장). 방치 시 운세 등급 하락.
*   **죽음과 도감:** 방치로 포만감이 0이 된 상태가 이틀 지속되면 펫은 '별'이 되어 **도감(Memorial)**으로 이동하고, 새 알을 즉시 뽑습니다.
*   **명예의 전당 랭킹:** 랭킹 보드에는 현재 키우는 펫뿐만 아니라 "역대 누적 돌봄 점수 최고 펫" 랭킹을 두어 펫 사망 시의 상실감을 방어합니다.
*   **MBTI 정규화:** '절대량'이 아닌 '비율과 패턴(세션 간격, 새것 시도 등)'으로 성격을 판정해 라이트 유저도 다양한 MBTI 획득 가능.

### 4-B. 운세 및 조합 규모
*   **첫 탭 락(Lock) 및 구제:** 하루 첫 확인 시점의 스탯으로 그날 운세 고정. 락 이후 첫 밥주기 1회에 한해 최종 등급 +1 상향(최대 5).
*   **사주 조합 규모:** 일간(본질 10종) × 월지(환경 4~12종) × 5등급 = 약 200~600개 베이스 텍스트. Gemini 배치 생성 비용 최적화 사이즈.
